The diagnosis

The web was built
for documents

The metaphors are extensions of the printed page.

Early websites looked like print. Some still do.

But YouTube, Discord, Bluesky — these are not documents.
90s

HTML + CSS + JS

Document object model. Print metaphors. Separation of concerns for static content.

00s

Dynamic elements added

JS grew the DOM API. Ajax arrived. Sites started behaving like apps — awkwardly.

10s

Framework explosion

React, Angular, Vue invented the component model on top of the document model. Styling frameworks appeared. Markup, style, and behavior merged — violating the separation of concerns the standards were built around.

now

Rust → WASM → JS bindings → DOM

Developers would rather write Rust that compiles to WASM and generates JS bindings just to avoid using JS directly.

The key insight

Two abstractions.
Neither produces the other.

Document abstraction
Static hierarchy of content
Implicit layout and styling
Concerns separated across files
No native statefulness
⊥
Component abstraction
Reusable, stateful, self-contained
Explicit behavior by design
Markup, style, behavior together
Interactivity is first-class
These are orthogonal. The browser has been awkwardly expanding one toward the other for 30 years.
No amount of expansion of one abstraction naturally produces the other.
The vision

BFTML

A new set of web standards that starts from the component abstraction — not as a destination the platform is expanding toward, but as the foundation from the beginning.
A modern markup language designed and optimized for making web apps. Not documents that happen to behave like apps.
// A button in BFTML
<event onclick=subscribe()>
  <box color=red>
    <text color=white>
      SUBSCRIBE
    </text>
  </box>
</event>
Every element has exactly one responsibility. Behavior, layout, style, and content are separate primitives — composed explicitly, not implied.
Design principles

Built on four pillars

01 — Correct abstraction

Component-first

Web apps deserve their own standards, not a document platform that's been awkwardly extended for three decades.

02 — Single responsibility

One job per element

<box> draws a box. <outline> draws an outline. <event> handles input. Each primitive does exactly one thing.

03 — Consistent syntax

No surprises

All tags follow the same grammar. Self-closing requires explicit syntax — always. Two stages of errors: syntax errors and tag errors. No silent ambiguity.

04 — Explicitness

Loud by default

No implicit defaults hiding bugs. Invalid code renders visible error regions — not dysfunctional silent output. Fallbacks must be explicitly opted into.

The runtime

Write components
in any language

BFTML provides a WASM-native runtime that can directly manipulate markup — no JS bridge required.
Components are scoped to their subtree. No global DOM access. The safety and structure of a component framework, without the lock-in of a single language.

Primitive elements

Compiled to WASM

The foundational layer — <box>, <text>, <row>, <grid> — compiled for performance. GPU access via WASI interface. Custom primitives supported.

Component layer

Also WASM

Components composed from primitives are also WASM modules. No interpreted layer, no runtime language lock-in — the same model applies at every level of the stack.

The runtime

Reactive by design,
cheap by default

BFTML's render model has three phases, each with a distinct job.
Phase 1 — load

Static tree construction

Components are stitched into a single element tree. This structure persists for the lifetime of the app — it is never rebuilt.

Phase 2 — state change

Push-reactive dirty flags

When state changes, affected nodes are immediately marked dirty. No polling. No scheduled diffing. The flag is set at the moment of change.

Phase 3 — render

Lazy traversal + cached renderers

At render time the tree is traversed. Clean nodes reuse their cached Renderer — a fully assembled pipeline, instance buffer, and bind groups — and call render() directly. Dirty nodes rebuild and cache.

The developer experience

Debug the app,
not the runtime

WASM is an implementation detail. You should never have to see it in your dev tools.
All behavior in BFTML routes through signals. That's what the dev tools surfaces — not bytecode, not a virtual DOM diff, not a framework's internal state tree.
The abstraction is clean enough that the debug surface matches how you reason about your app.
Signal graph

See the reactive topology

Visualize which signals exist, what they depend on, and how they connect — the live wiring diagram of your app's reactive state.

Dirty flag propagation

Watch state flow in real time

As state changes, see exactly which nodes get marked dirty, in what order, and which draw commands get recomputed as a result.

The principle

One abstraction, all the way down

Signals are how you write behavior. Signals are how you debug behavior. The layer you think in and the layer you inspect are the same layer.

The ecosystem

A decentralized
component registry

Package registries are centralized choke points. BFTML's component ecosystem is built on ATProto — the same protocol that powers Bluesky.
Developers publish components to their own personal data server. No single company owns the registry. No single point of failure. Your components live where your data lives — with you.
AppViews aggregate those components and serve them in the same way you'd use npm or crates.io today — searchable, versioned, installable. Except the underlying data is open and portable.
Personal Data Server (PDS)

You own your components

Every developer publishes to their own PDS — a server they control. Components are records in your personal data repository, signed with your identity. No platform can unpublish them.

AppView

Aggregation without lock-in

AppViews crawl the network of PDSes and index components into a searchable, browsable registry — like npm, but built on open data anyone can re-aggregate. Competing AppViews can coexist.

The pattern

Familiar interface, open infrastructure

Install components the same way you would today. The difference is what's underneath — a federated network of personal servers rather than a corporate-owned registry with terms of service.
